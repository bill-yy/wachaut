# Pre-lanzamiento — 7 mejoras

## 1. Quitar console.log spam
**Archivo:** `apps/web/src/lib/sfu-client.ts`, `apps/web/src/routes/room/+page.svelte`

- Gatear todos los `console.log('[sfu]...')` con `import.meta.env.DEV`:
  ```ts
  if (import.meta.env.DEV) console.log('[sfu] ...');
  ```
- O crear un helper `devlog()` que sea no-op en producción
- Eliminar los `console.error` que también son debug (mantener los que son errores reales)
- Impacto: consola limpia en producción, credibilidad instantánea

## 3. Página de privacidad
**Archivos:** nuevo `apps/web/src/routes/privacidad/+page.svelte`, `apps/web/src/routes/+page.svelte` (footer link)

- Página honesta que explique:
  - Qué datos procesamos: ninguno persistente. Todo es in-memory.
  - Las salas se eliminan al cerrarlas (no hay base de datos)
  - El PIN se hashea (SHA-256), nunca se almacena en claro
  - La transmisión va cifrada (DTLS-SRTP) a través de un servidor SFU
  - No usamos cookies de tracking, solo localStorage para preferencias (username, favoritos, tema)
  - No compartimos datos con terceros
  - Contacto
- Link en el footer del landing: "Privacidad"
- Diseño coherente con la app (dark, amber, glow)

## 4. Host refresh survival
**Archivo:** `apps/web/src/routes/room/+page.svelte`

- En `sessionStorage` guardar `{ roomId, pin }` con key `wachaut.host.session`
- Al montar el componente, si hay sesión guardada Y la sala sigue activa (el server no ha cerrado), reclamar la sala en vez de crear una nueva
- En `cleanup()` / `leaveRoom()`, limpiar el sessionStorage
- Flujo: host monta → ¿hay sesión? → re-emit `host:create-room` con el roomId guardado → server responde `room:created` → reclamar sala
- Si la sala ya no existe (server la cerró tras el grace timer), crear una nueva y avisar

## 5. Pill "Servicio activo" real
**Archivo:** `apps/web/src/routes/+page.svelte` (landing)

- Si responde OK → mostrar pill verde "Servicio activo"
- Si falla → mostrar pill rojo "Servicio temporalmente no disponible" o ocultar
- Re-check cada 60s
