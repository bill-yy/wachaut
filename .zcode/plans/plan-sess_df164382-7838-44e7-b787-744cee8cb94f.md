# Fix: /turn-credentials 404 — entregar TURN creds via Socket.IO

## Problema
El cliente hace `fetch('https://sfu-wachaut.billytech.es/turn-credentials')` → 404.
El SFU está desplegado con raw port mapping (sin labels Traefik), así que solo
WebSocket/Socket.IO llega al SFU. Las peticiones HTTP GET no se rutear.

## Solución: entregar TURN creds via Socket.IO (sin nuevo routing)

En vez de un endpoint HTTP separado que requiere configuración de Traefik/Dokploy,
entregar las credenciales TURN dentro del flujo Socket.IO que ya funciona.

### Cambios:

**1. SFU (`apps/sfu/src/index.ts`)**
- Eliminar el endpoint HTTP `/turn-credentials`
- En el handler `join-room`, incluir `iceServers` en la respuesta del callback
  (junto con `rtpCapabilities`, `peers`, etc.)
- El SFU ya tiene `TURN_URL`, `TURN_SECRET` y `generateTurnCredentials()`

**2. Cliente (`apps/web/src/lib/sfu-client.ts`)**
- Eliminar `#fetchIceServers()` (el fetch HTTP que falla)
- En `joinRoom()`, leer `iceServers` de la respuesta del callback de `join-room`
- Guardarlos en `#iceServers` para que `produce()` y `consume()` los usen

### Ventajas
- **Cero cambios de infraestructura** — no necesita tocar Traefik/Dokply/compose
- Funciona con el deploy actual
- Más eficiente: una sola conexión (Socket.IO) en vez de dos (Socket.IO + HTTP fetch)
- Elimina la race condition del fetch asíncrono
